HTTP-сервис для бронирования переговорных комнат внутри компании. Администраторы создают переговорки и настраивают расписание их доступности; система сама формирует 30-минутные слоты на основе расписания. Сотрудники просматривают свободные слоты, создают и отменяют брони.

---

## Быстрый старт

```bash
docker-compose up -d --build
```

Сервис: `http://localhost:8080` | Swagger UI: `http://localhost:8080/swagger/`

---

## Использование

```bash
make up               # запустить сервис со всеми зависимостями
make down             # остановить и удалить контейнеры
make seed             # наполнить БД тестовыми данными
make test             # все тесты
make test-unit        # только unit-тесты (без БД)
make test-e2e         # E2E-тесты через docker-compose
make coverage         # покрытие unit-тестами (82%)
make swagger          # регенерировать Swagger-документацию
make mock             # регенерировать моки
make lint             # запустить линтеры
```

## Целевой масштаб

Сервис проектировался под корпоративную компанию среднего размера со следующим профилем нагрузки:

| Параметр                          | Значение  |
|-----------------------------------|-----------|
| Переговорных комнат               | до 50     |
| Сотрудников                       | до 10 000 |
| Слотов в день                     | до 1 000  |
| Активных броней                   | до 100 000|
| RPS                               | 100       |
| SLI успешности                    | 99.9%     |
| SLI времени ответа p99 (слоты)    | 200 мс    |
| Запросы слотов на ближайшие 7 дней| ~99.9%    |

Самый нагруженный эндпоинт — `GET /rooms/{roomId}/slots/list`: именно под него в первую очередь оптимизировались хранение и индексы. Этот профиль определил выбор гибридной генерации слотов, горизонт lazy-генерации и параметры пула соединений.

---

## Принятые решения

### Генерация слотов

Выбран **гибридный подход** (Eager + Lazy generation):

- **Eager (жадная генерация):** при создании расписания система _сразу_ генерирует все слоты на **следующие 7 дней** и сохраняет их в БД. Запрос на получение слотов для этого диапазона — чистый `SELECT` без вычислений.
- **Lazy (ленивая генерация):** при запросе слотов на дату _за пределами_ 7 дней слоты вычисляются и сохраняются в БД в момент запроса, а не заранее. Lazy-генерация ограничена горизонтом **90 дней** от текущей даты.

> **Обоснование:**
>
> **99.9%** запросов приходится на ближайшие 7 дней — для них слоты уже готовы в БД, `GET /rooms/{roomId}/slots/list` работает как чистый `SELECT` с индексом по `(room_id, start_time)`.
> Eager-генерация только на 7 дней вперёд минимизирует нагрузку на БД при создании расписания. Lazy-генерация для дальних дат обеспечивает корректность без постоянного фонового джоба.
>
> Ограничение в 90 дней для Lazy-генерации — самостоятельное архитектурное решение.
> Без ограничения любой запрос на произвольную дату в будущем (например, 2099 год) генерировал бы и сохранял слоты, что приводит к бесконтрольному росту БД.
> 90 дней выбрано как разумный деловой горизонт планирования переговорок: бронирование переговорной комнаты более чем на 3 месяца вперёд — нетипичный сценарий для корпоративного использования.

### Конференц-ссылка при сбоях внешнего сервиса

При передаче `createConferenceLink: true` система вызывает Conference Service до сохранения брони в БД. Если сервис вернул ссылку, она включается в единственный `INSERT` вместе с бронью.

**Смоделированные сценарии сбоев:**

- **Сценарий 1: Conference Service недоступен (ошибка сети, таймаут)**
  Conference Service возвращает ошибку, бронь создаётся без ссылки (`conferenceLink = null`), ошибка логируется через `slog.Warn`. Бронирование не блокируется — недоступность второстепенного сервиса не должна блокировать критичную операцию.

- **Сценарий 2: Conference Service вернул ссылку, затем `INSERT` брони упал**
  Реализован подход **идемпотентного ключа**: `Stub` хранит маппинг `bookingID -> link`, повторный вызов с тем же `bookingID` возвращает ту же ссылку. Retry при падении `INSERT` безопасен.

### Идемпотентность отмены брони

Повторный вызов `POST /bookings/{bookingId}/cancel` на уже отменённой брони возвращает `200 OK` с актуальным состоянием (`status: cancelled`).
